이 구조는 **레이어드 아키텍처(계층형 구조)**를 따릅니다. 하나의 요청이 `apis → schemas → services → repositories → models` 순서로 흘러가며, 각 계층은 자기 역할만 책임집니다. 예를 들어 "환자 등록" 요청이 들어오면 apis가 요청을 받아 → schemas가 데이터 형태를 검증하고 → services가 "이미 등록된 환자인지" 같은 업무 규칙을 확인한 뒤 → repositories가 실제 DB에 INSERT 쿼리를 실행합니다.

## 1. 각 디렉터리는 어떤 역할을 수행하며, 어떤 파일을 작성해야 하는가

### `app/core/`
앱 전역 설정, DB 연결, 환경변수 등 **프로젝트 공통 뼈대**를 모아두는 곳입니다. 다른 모든 계층이 이 폴더의 설정을 참조합니다.
- `config.py` — DB 접속 주소, 비밀 키 등 환경변수를 코드 전체에서 재사용할 수 있게 정의
- `db/databases.py` — DB 엔진(`engine`)과 세션 팩토리(`SessionLocal`)를 정의
- `db/models.py` — 모든 ORM 모델이 상속받는 `Base` 클래스를 정의

### `app/models/`
SQLAlchemy ORM 모델, 즉 **DB 테이블의 설계도**를 정의하는 곳입니다. ERD에 그려진 테이블 하나당 파일 하나를 만드는 것이 일반적입니다.
- `patient.py` — 환자(Patient) 테이블 모델
- `doctor.py` — 의사(Doctor) 테이블 모델
- `visit.py` — 진료기록(Visit), 처방(Prescription) 테이블 모델

### `app/repositories/`
DB에 **직접 접근**해 데이터를 조회·저장·수정·삭제하는 쿼리 코드를 모아두는 곳입니다. SQL 또는 ORM 쿼리는 오직 이 계층에서만 작성합니다.
- `patient_repository.py` — 예: `get_patient_by_id()`, `create_patient()`, `update_patient()`

### `app/schemas/`
API로 주고받는 데이터의 **형태(DTO, Pydantic 모델)**를 정의하는 곳입니다. 요청용과 응답용 스키마를 분리해서 작성합니다.
- `patient_schema.py` — 예: `PatientCreateRequest`, `PatientUpdateRequest`, `PatientResponse`

### `app/services/`
실제 **업무 로직(비즈니스 로직)**을 처리하는 곳입니다. "환자 등록 시 전화번호 중복 확인 후 저장" 같은 절차가 여기 들어갑니다. apis와 repositories 사이에서 규칙을 판단하는 역할입니다.
- `patient_service.py` — 예: 중복 검증, 나이 계산, 상태 변경 규칙

### `app/apis/`
HTTP 요청을 받는 **라우터(Controller)**입니다. URL 경로와 처리 함수를 연결하는 역할만 하고, 실제 로직은 services에 위임합니다.
- `patient_apis.py` — 예: `GET /patients`, `POST /patients`, `PATCH /patients/{patient_id}`

> **비유** — 병원 접수 과정
> 환자가 접수창구(**apis**)에 서류를 내면 → 접수 담당자가 서류 양식(**schemas**)을 확인하고 → 간호사가 업무 절차(**services**)를 진행하고 → 의무기록실(**repositories**)이 실제 캐비닛(**models**가 정의한 DB 테이블)에 서류를 넣거나 꺼냅니다. 역할을 층(layer)으로 나누면 한 곳을 고쳐도 다른 곳이 덜 흔들립니다.

## 2. 각 파일의 역할

| 파일 | 역할 |
|---|---|
| `app/main.py` | FastAPI 앱을 생성하고, 각 `apis` 라우터를 등록해 실제 서버로 실행하는 진입점(entry point) |
| `app/core/config.py` | DB 접속 주소, 비밀 키 등 환경변수를 코드 전체에서 쓸 수 있도록 모아두는 설정 파일 |
| `pyproject.toml` | 프로젝트 이름, 파이썬 버전, 설치할 라이브러리(FastAPI, SQLAlchemy 등) 목록을 적어두는 설정 파일 |
| `uv.lock` | `pyproject.toml`에 적힌 라이브러리들의 정확한 버전을 고정해 팀원 전원이 같은 버전을 쓰게 만드는 잠금 파일 (직접 수정하지 않고 `uv add`/`uv sync`로만 갱신) |

## 3. 데이터베이스 연결 설정 (`app/core/db/databases.py`)

```python
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

# 1. DB 연결 설정 (엔진 생성) — 병원 진료기록 DB
DATABASE_URL = "sqlite:///./clinic.db"
engine = create_engine(
    DATABASE_URL,
    connect_args={"check_same_thread": False},
)

# 2. 세션 팩토리 정의 → 관례적으로 변수명을 'SessionLocal'로 지정
SessionLocal = sessionmaker(
    autocommit=False,
    autoflush=False,
    bind=engine,
)

# 3. FastAPI 의존성 주입용 함수
def get_db():
    db = SessionLocal()   # 세션(연결) 하나를 생성
    try:
        yield db          # 요청 처리 로직에 세션을 빌려줌
    finally:
        db.close()        # 요청이 끝나면 반드시 세션을 닫음
```

### SessionFactory vs SessionLocal
- **SessionFactory** — 세션을 필요할 때마다 찍어내는 "설계도(공장)"라는 일반 개념. `sessionmaker()`가 이 팩토리를 만들어준다.
- **SessionLocal** — `sessionmaker()`로 만든 실제 팩토리 인스턴스의 관례적 이름. `SessionLocal()`을 호출해야 실제로 쓸 수 있는 세션(`db`)이 만들어진다. (붕어빵 틀 vs 붕어빵의 관계)

### 세션 옵션 3대장 (모두 `False`로 두는 이유)

| 옵션 | False로 쓰는 이유 (환자 등록 예시) |
|---|---|
| `autocommit` | "환자 등록 → 진료기록 생성"처럼 여러 단계가 하나의 트랜잭션일 때, 중간에 에러가 나면 전부 되돌려야(Rollback) 하므로 수동 커밋이 안전하다 |
| `autoflush` | 조회할 때마다 자동으로 DB와 통신하지 않도록 막아, 완벽히 검증된 데이터만 DB로 흘려보낼 수 있다 |
| `expire_on_commit` | 커밋 후에도 방금 만든 객체 값을 DB 재조회 없이 그대로 읽을 수 있어 불필요한 SELECT를 줄인다 |

### `session.close()`를 반드시 써야 하는 이유
- **커넥션 풀 반납** — 닫지 않으면 DB 연결 통로가 꽉 차서 새 요청이 대기하게 됨
- **메모리 누수 방지** — 조회해온 데이터가 메모리에 계속 쌓이는 것을 막음
- **Lock 해제** — 수정 도중 에러가 나도 되돌리기와 잠금 해제를 보장

## 4. SQLAlchemy 모델 작성 & Alembic 마이그레이션 절차

### 4-1. 모델 작성 예시 (`app/models/patient.py`)

```python
from datetime import date, datetime

from sqlalchemy import String, Date, DateTime, func
from sqlalchemy.orm import Mapped, mapped_column, relationship

from app.core.db.models import Base


class Patient(Base):
    __tablename__ = "patients"

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(50), index=True)
    phone: Mapped[str] = mapped_column(String(20), unique=True, index=True)
    birth_date: Mapped[date] = mapped_column(Date)
    created_at: Mapped[datetime] = mapped_column(
        DateTime, server_default=func.now()
    )

    visits: Mapped[list["Visit"]] = relationship(
        back_populates="patient",
        cascade="all, delete-orphan",
    )
```

- `primary_key=True` — 각 환자를 대표하는 식별자(기본키). 중복·NULL 불가, 테이블당 1개만 지정.
- `unique=True` — 같은 전화번호로 환자가 중복 등록되는 것을 방지(고유키). 기본키와 달리 여러 개 지정 가능하고 NULL 허용.
- `ForeignKey` + `ON DELETE CASCADE` — 자식 테이블(예: `prescriptions.visit_id`)이 부모 테이블(`visits.id`)을 가리키게 하고, 부모가 삭제되면 자식도 함께 삭제되도록 지정.

### 4-2. Alembic 마이그레이션 절차

| 순서 | 명령/작업 | 설명 |
|---|---|---|
| 1 | `app/models/`에 모델 파일 작성 | 이 시점에는 아직 실제 DB에 변화 없음 (파이썬 코드일 뿐) |
| 2 | `alembic revision --autogenerate -m "add patient table"` | 모델 코드와 실제 DB 상태의 차이를 비교해 마이그레이션 파일을 `alembic/versions/`에 자동 생성 |
| 3 | 생성된 마이그레이션 파일 검토 | `upgrade()`/`downgrade()` 함수가 의도한 컬럼·제약조건과 일치하는지 사람이 직접 확인 |
| 4 | `alembic upgrade head` | 검토를 마친 마이그레이션을 실제 데이터베이스에 적용 (이때 비로소 테이블이 생성됨) |
| 5 | DB Viewer로 확인 | 테이블·컬럼이 ERD와 일치하는지 확인하고 스크린샷을 남김 (`docs/3일차_db_migration.md`) |
| 6 | PR 생성 & 리뷰 & 병합 | 모델·마이그레이션 파일을 함께 커밋해 PR을 만들고 팀원 리뷰 후 main/develop에 병합 |

## 5. API 구현 흐름

1. **요청 수신** — 클라이언트가 `POST /patients`로 JSON(`{"name": "홍길동", "phone": "010-1234-5678"}`)을 보내면 `apis` 계층이 요청을 받는다.
2. **데이터 검증** — `schemas`의 `PatientCreateRequest`가 JSON을 Pydantic 객체로 변환하며 필수값·형식(전화번호 패턴 등)을 자동 검증한다. FastAPI가 JSON ↔ 파이썬 객체 변환을 자동으로 처리해주기 때문에 `body.name`처럼 바로 값을 꺼낼 수 있다.
3. **업무 로직 처리** — `services`가 전화번호 중복 여부 등 업무 규칙을 확인한다.
4. **DB 반영** — `repositories`가 `models`로 정의된 테이블에 실제 INSERT 쿼리를 실행한다.
5. **응답 반환** — 저장된 결과를 `PatientResponse` 스키마로 변환해 클라이언트에 JSON으로 반환한다. `response_model`에 맞지 않는 임의의 객체는 리턴할 수 없다.

### 참고 — 등록(Create) vs 수정(Update) API 설계 원칙

| 구분 | POST (등록) | PATCH (부분 수정) |
|---|---|---|
| 필수값 | 대부분 필수 | 대부분 선택(Optional) — 바꿀 필드만 보내면 됨 |
| 대상 없음 | 새로 생성 | 일반적으로 404 |
| 스키마 | `PatientCreateRequest` | `PatientUpdateRequest` (모든 필드 `None` 기본값) |

수정 로직에서 여러 필드를 동시에 반영할 때는 필드끼리 서로 무관하므로 `if - if` 패턴을 사용한다 (예: 이름과 전화번호를 동시에 수정). 반대로 여러 조건 중 하나만 적용해야 하는 경우(예: 응급실 중증도 등급 판정)에는 `if - elif`를 사용한다.
