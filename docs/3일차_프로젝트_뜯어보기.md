# 3일차 프로젝트 템플릿 뜯어보기

## 1. 프로젝트 구조를 먼저 이해해야 하는 이유

이 프로젝트는 FastAPI 애플리케이션을 기능별 계층으로 나누어 관리하도록 구성되어 있다.

작은 예제에서는 API 함수 안에 검증, 비즈니스 로직, 데이터베이스 접근을 모두 작성할 수 있다. 하지만 기능이 늘어나면 한 파일이 지나치게 커지고 같은 코드가 반복된다. 이를 막기 위해 역할에 따라 코드를 나눈다.

이 프로젝트에서 권장하는 기본 흐름은 다음과 같다.

```text
클라이언트 요청
    ↓
API 라우터(app/apis)
    ↓
요청 데이터 검증(app/schemas)
    ↓
비즈니스 로직(app/services)
    ↓
데이터베이스 접근(app/repositories)
    ↓
SQLAlchemy 모델(app/models)
    ↓
MySQL
```

공통 설정과 데이터베이스 연결은 `app/core`에서 담당한다.

```text
app/core/config.py
    ↓ 환경 변수 읽기
app/core/db/databases.py
    ↓ 엔진, 세션, Base 제공
models / repositories / services / apis
```

계층을 나누는 목적은 파일 수를 늘리는 것이 아니라 각 코드의 책임을 명확하게 만드는 것이다.

## 2. 주요 디렉터리의 역할

### 2.1 `app/core/`

애플리케이션 전체에서 공통으로 사용하는 기반 설정을 관리한다.

현재 템플릿에는 다음 파일이 있다.

```text
app/core/
├── __init__.py
├── config.py
└── db/
    ├── __init__.py
    ├── databases.py
    └── models.py
```

주요 역할은 다음과 같다.

- 환경 변수와 애플리케이션 설정 관리
- 데이터베이스 엔진 및 세션 생성
- SQLAlchemy 모델이 공통으로 사용할 `Base` 제공
- UUID, 생성일, 수정일, 소프트 삭제와 같은 공통 모델 기능 제공
- 프로젝트가 커지면 인증, 로깅, 공통 예외 처리, 보안 설정 등을 추가할 수 있음

`core`에는 회원 가입, 게시글 작성과 같은 특정 기능의 비즈니스 로직을 넣지 않는다. 여러 기능에서 공통으로 사용할 기반 코드만 둔다.

#### 현재 `config.py`의 역할

`pydantic-settings`의 `BaseSettings`를 사용해 `.env`의 데이터베이스 정보를 읽는다.

현재 설정 항목은 다음과 같다.

- `DB_USER`
- `DB_PASSWORD`
- `DB_HOST`
- `DB_PORT`
- `DB_NAME`

`.env`에 값이 없을 때는 코드에 작성된 기본값을 사용한다.

#### 현재 `db/models.py`의 역할

여러 SQLAlchemy 모델에서 반복되는 컬럼을 믹스인으로 제공한다.

- `UUIDMixin`: UUID 기본 키
- `TimestampMixin`: 생성일과 수정일
- `SoftDeleteMixin`: 실제 행을 삭제하지 않고 삭제 상태를 표시하는 컬럼

새 모델을 작성할 때 필요한 믹스인을 선택해 상속할 수 있다.

### 2.2 `app/models/`

데이터베이스 테이블과 연결되는 SQLAlchemy ORM 모델을 작성한다.

모델에는 다음 내용을 작성한다.

- 테이블 이름
- 기본 키
- 컬럼 이름과 자료형
- `nullable`, `unique`, `index` 같은 제약 조건
- 외래 키
- 테이블 간 관계
- 공통 믹스인 상속

예를 들어 회원 테이블은 다음과 같이 작성할 수 있다.

```python
# app/models/user.py
from sqlalchemy import Integer, String
from sqlalchemy.orm import Mapped, mapped_column

from app.core.db.databases import Base
from app.core.db.models import TimestampMixin


class User(TimestampMixin, Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(
        Integer,
        primary_key=True,
        autoincrement=True,
    )
    name: Mapped[str] = mapped_column(
        String(10),
        nullable=False,
    )
    age: Mapped[int] = mapped_column(
        Integer,
        nullable=False,
    )
    email: Mapped[str] = mapped_column(
        String(30),
        nullable=False,
        unique=True,
        index=True,
    )
    password: Mapped[str] = mapped_column(
        String(255),
        nullable=False,
    )
```

실제 서비스에서는 비밀번호 원문을 저장하면 안 된다. 비밀번호는 서비스 계층에서 안전한 해시로 변환한 뒤 저장해야 한다.

#### Alembic이 모델을 찾게 만드는 방법

현재 `alembic/env.py`는 다음 import를 통해 모델을 읽는다.

```python
from app import models
```

따라서 새 모델 파일을 만들기만 해서는 Alembic이 모델을 발견하지 못할 수 있다. `app/models/__init__.py`에서 모델을 import해야 한다.

```python
# app/models/__init__.py
from app.models.user import User

__all__ = ["User"]
```

현재 템플릿의 `app/models/__init__.py`는 비어 있으므로, 첫 모델을 만들 때 반드시 이 부분도 함께 작성해야 한다.

### 2.3 `app/repositories/`

데이터베이스에 직접 접근하는 코드를 작성한다.

Repository는 SQLAlchemy의 `AsyncSession`을 사용하여 다음 작업을 수행한다.

- 데이터 조회
- 데이터 추가
- 데이터 수정
- 데이터 삭제
- 조건 검색
- 정렬과 페이지네이션

예시는 다음과 같다.

```python
# app/repositories/user_repository.py
from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession

from app.models.user import User


class UserRepository:
    def __init__(self, db: AsyncSession):
        self.db = db

    async def get_by_id(self, user_id: int) -> User | None:
        return await self.db.get(User, user_id)

    async def get_by_email(self, email: str) -> User | None:
        result = await self.db.execute(
            select(User).where(User.email == email)
        )
        return result.scalar_one_or_none()

    async def create(self, user: User) -> User:
        self.db.add(user)
        await self.db.flush()
        await self.db.refresh(user)
        return user

    async def delete(self, user: User) -> None:
        await self.db.delete(user)
        await self.db.flush()
```

Repository에서는 가능한 한 HTTP 상태 코드나 FastAPI의 `HTTPException`을 직접 다루지 않는다. Repository는 데이터베이스 작업에 집중하고, 존재하지 않는 회원을 404로 응답할지는 API 또는 Service 계층에서 판단한다.

`commit()`과 `rollback()`의 위치는 팀 규칙으로 정해야 한다. 이 프로젝트에서는 여러 Repository 작업을 하나의 업무 단위로 묶을 수 있도록 Service 계층에서 트랜잭션을 완료하는 방식을 권장한다.

### 2.4 `app/schemas/`

API 요청과 응답의 데이터 형식을 Pydantic 모델로 정의한다.

주요 역할은 다음과 같다.

- Request Body 검증
- 응답에 포함할 필드 제한
- 필수값과 선택값 구분
- 문자열 길이와 숫자 범위 검증
- 이메일과 비밀번호 같은 값의 형식 검증
- SQLAlchemy 객체를 API 응답으로 변환

예시는 다음과 같다.

```python
# app/schemas/user.py
from pydantic import BaseModel, ConfigDict, Field


class UserCreate(BaseModel):
    name: str = Field(min_length=2, max_length=10)
    age: int = Field(ge=14)
    email: str = Field(max_length=30)
    password: str = Field(min_length=8, max_length=20)


class UserUpdate(BaseModel):
    age: int | None = Field(default=None, ge=14)
    email: str | None = Field(default=None, max_length=30)
    password: str | None = Field(
        default=None,
        min_length=8,
        max_length=20,
    )


class UserResponse(BaseModel):
    model_config = ConfigDict(from_attributes=True)

    id: int
    name: str
    age: int
    email: str
```

`UserCreate`, `UserUpdate`, `UserResponse`를 분리하는 이유는 요청과 응답에서 필요한 값이 다르기 때문이다.

예를 들어 비밀번호는 회원 생성 요청에는 필요하지만 응답에는 포함하면 안 된다.

SQLAlchemy 객체를 응답 모델로 변환할 때는 Pydantic v2의 `ConfigDict(from_attributes=True)`를 사용할 수 있다.

### 2.5 `app/services/`

서비스의 실제 업무 규칙을 구현한다.

Service 계층은 API와 Repository 사이에서 다음 역할을 수행한다.

- 중복 이메일 확인
- 비밀번호 해시 처리
- 회원 가입 조건 판단
- 여러 Repository 작업 조합
- 트랜잭션 commit 및 rollback
- 업무 규칙에 따른 예외 발생

예시는 다음과 같다.

```python
# app/services/user_service.py
from sqlalchemy.ext.asyncio import AsyncSession

from app.models.user import User
from app.repositories.user_repository import UserRepository
from app.schemas.user import UserCreate


class DuplicateEmailError(Exception):
    pass


class UserService:
    def __init__(self, db: AsyncSession):
        self.db = db
        self.repository = UserRepository(db)

    async def create_user(self, payload: UserCreate) -> User:
        existing_user = await self.repository.get_by_email(payload.email)

        if existing_user is not None:
            raise DuplicateEmailError("이미 등록된 이메일입니다.")

        user = User(
            name=payload.name,
            age=payload.age,
            email=payload.email,
            # 실제 구현에서는 해시된 비밀번호를 저장해야 한다.
            password=payload.password,
        )

        try:
            created_user = await self.repository.create(user)
            await self.db.commit()
            return created_user
        except Exception:
            await self.db.rollback()
            raise
```

API 함수에 모든 로직을 작성하면 다른 API나 백그라운드 작업에서 같은 기능을 재사용하기 어렵다. Service로 분리하면 회원 생성 규칙을 여러 곳에서 동일하게 사용할 수 있다.

### 2.6 `app/apis/`

FastAPI의 `APIRouter`를 사용하여 외부 요청을 받는 엔드포인트를 작성한다.

API 계층의 역할은 다음과 같다.

- URL과 HTTP Method 정의
- Path Parameter, Query Parameter, Request Body 수신
- Pydantic Schema를 통한 요청 검증
- `Depends`를 사용한 데이터베이스 세션 주입
- Service 호출
- HTTP 상태 코드와 응답 형식 지정
- 업무 예외를 적절한 HTTP 응답으로 변환

예시는 다음과 같다.

```python
# app/apis/users.py
from fastapi import APIRouter, Depends, HTTPException, status
from sqlalchemy.ext.asyncio import AsyncSession

from app.core.db.databases import async_get_db
from app.schemas.user import UserCreate, UserResponse
from app.services.user_service import DuplicateEmailError, UserService


router = APIRouter(
    prefix="/api/v1/users",
    tags=["users"],
)


@router.post(
    "",
    response_model=UserResponse,
    status_code=status.HTTP_201_CREATED,
)
async def create_user(
    payload: UserCreate,
    db: AsyncSession = Depends(async_get_db),
):
    service = UserService(db)

    try:
        return await service.create_user(payload)
    except DuplicateEmailError as error:
        raise HTTPException(
            status_code=status.HTTP_409_CONFLICT,
            detail=str(error),
        ) from error
```

라우터를 만든 뒤에는 `app/main.py`에서 등록해야 Swagger UI에 표시된다.

```python
from app.apis import users

app.include_router(users.router)
```

## 3. 주요 파일의 역할

### 3.1 `app/main.py`

FastAPI 애플리케이션의 시작점이다.

현재 템플릿에서는 다음 역할을 수행한다.

- `FastAPI()` 인스턴스 생성
- 정적 파일 디렉터리 마운트
- 미디어 파일 디렉터리 마운트
- API 라우터 등록
- `/healthcheck` 제공
- 루트 페이지 제공
- 등록되지 않은 프론트엔드 경로를 `index.html`로 보내는 catch-all 처리

서버 실행 명령의 `app.main:app`은 다음 의미이다.

```text
app.main  → app/main.py 모듈
app       → main.py 안의 FastAPI 인스턴스 변수
```

실행 예시는 다음과 같다.

```bash
uv run fastapi dev app/main.py
```

또는 다음과 같이 실행할 수 있다.

```bash
uv run uvicorn app.main:app --reload
```

라우터는 catch-all 라우트보다 먼저 등록해야 한다. 그렇지 않으면 API 요청이 프론트엔드용 `index.html` 응답으로 처리될 수 있다.

현재 코드의 `BASE_DIR`은 `app`의 상위인 프로젝트 루트를 가리킨다. 따라서 실제 정적 파일은 프로젝트 루트의 `static/`, `media/`에서 사용되고 있다. 과제 안내 구조의 `app/static/`, `app/media/`와 차이가 있으므로 팀에서 실제 기준 경로를 하나로 통일해야 한다.

### 3.2 `app/core/config.py`

환경 변수를 Python 객체로 읽는 설정 파일이다.

현재 템플릿은 다음과 같이 동작한다.

```python
class Settings(BaseSettings):
    DB_USER: str = "root"
    DB_PASSWORD: str = "password1234"
    DB_HOST: str = "localhost"
    DB_PORT: str = "3306"
    DB_NAME: str = "ai_health"
```

그리고 다음 설정으로 프로젝트 루트의 `.env`를 읽는다.

```python
model_config = {
    "env_file": ".env",
    "extra": "ignore",
}
```

`.env` 예시는 다음과 같다.

```dotenv
DB_ROOT_PASSWORD=root-password
DB_USER=app-user
DB_PASSWORD=app-password
DB_HOST=127.0.0.1
DB_PORT=3306
DB_NAME=ai_health
```

Docker Compose 내부에서 FastAPI 컨테이너가 MySQL 컨테이너에 접근할 때는 `localhost`가 아니라 MySQL 서비스 이름을 사용해야 한다.

```dotenv
DB_HOST=mysql
```

정리하면 다음과 같다.

- FastAPI를 로컬에서 실행하고 MySQL 포트를 호스트에 연결한 경우: `127.0.0.1` 또는 `localhost`
- FastAPI와 MySQL을 모두 Docker Compose로 실행하는 경우: `mysql`

`.env`에는 비밀번호가 포함되므로 Git에 커밋하지 않는다. 현재 `.gitignore`에는 `.env`와 `*.env`가 제외 대상으로 등록되어 있다.

운영 환경에서는 코드에 작성된 기본 비밀번호에 의존하지 말고 필요한 환경 변수의 입력을 강제하는 것이 안전하다.

### 3.3 `pyproject.toml`

Python 프로젝트의 기본 정보와 직접 의존성을 선언하는 파일이다.

현재 프로젝트에는 다음 정보가 들어 있다.

- 프로젝트 이름: `ai-health-web-assignment`
- 프로젝트 버전: `0.1.0`
- Python 요구 버전: 3.13 이상
- FastAPI
- SQLAlchemy 비동기 기능
- Alembic
- MySQL 비동기 드라이버 `asyncmy`
- 환경 변수 설정용 `pydantic-settings`
- UUID7 생성용 `uuid6`

새 패키지는 파일을 직접 수정하기보다 uv 명령으로 추가하는 편이 안전하다.

```bash
uv add 패키지명
```

개발 환경에서만 사용할 도구는 개발 의존성으로 추가할 수 있다.

```bash
uv add --dev pytest
```

의존성을 설치하고 가상 환경을 맞출 때는 다음 명령을 사용한다.

```bash
uv sync
```

### 3.4 `uv.lock`

`pyproject.toml`에 선언된 직접 의존성과 그 하위 의존성의 정확한 버전을 기록한다.

예를 들어 `pyproject.toml`에는 다음처럼 최소 버전이 작성될 수 있다.

```toml
"alembic>=1.18.4"
```

`uv.lock`에는 실제로 선택된 패키지 버전, 다운로드 위치, 해시, 하위 의존성이 기록된다.

이 파일이 필요한 이유는 팀원과 배포 환경이 가능한 한 같은 패키지 조합을 설치하게 하기 위해서이다.

팀 규칙은 다음과 같이 정한다.

- `pyproject.toml`과 `uv.lock`은 함께 커밋
- `uv.lock`을 임의로 직접 편집하지 않음
- 패키지 추가와 삭제는 `uv add`, `uv remove` 사용
- 충돌이 발생하면 내용을 임의로 합치기보다 `uv lock` 또는 `uv sync`로 다시 생성 후 검증

`uv.lock`은 `.venv`와 달리 저장소에 커밋해야 한다.

## 4. 데이터베이스 연결 설정

### 4.1 `app/core/db/databases.py`의 구성

현재 템플릿은 비동기 SQLAlchemy와 MySQL을 사용한다.

#### 1단계: 데이터베이스 URL 생성

```python
DATABASE_PREFIX = "mysql+asyncmy://"
DATABASE_URI = (
    f"{settings.DB_USER}:{settings.DB_PASSWORD}"
    f"@{settings.DB_HOST}:{settings.DB_PORT}/{settings.DB_NAME}"
)
DATABASE_URL = f"{DATABASE_PREFIX}{DATABASE_URI}"
```

URL 형식은 다음과 같다.

```text
mysql+asyncmy://사용자:비밀번호@호스트:포트/데이터베이스명
```

- `mysql`: SQLAlchemy가 사용할 데이터베이스 종류
- `asyncmy`: 비동기 MySQL 드라이버

#### 2단계: 비동기 엔진 생성

```python
async_engine = create_async_engine(
    DATABASE_URL,
    echo=False,
    future=True,
)
```

Engine은 SQLAlchemy와 데이터베이스 사이의 연결 기반을 관리한다.

`echo=True`로 바꾸면 실행되는 SQL을 확인하기 쉽지만 로그에 민감한 정보가 포함될 수 있으므로 개발 중에만 제한적으로 사용한다.

#### 3단계: 세션 팩토리 생성

```python
AsyncSessionLocal = async_sessionmaker(
    bind=async_engine,
    autoflush=False,
    expire_on_commit=False,
)
```

API 요청마다 `AsyncSessionLocal`에서 세션을 하나 생성한다.

- `autoflush=False`: SQL 실행 시점을 명시적으로 관리
- `expire_on_commit=False`: commit 이후에도 객체 속성에 접근하기 쉽게 설정

#### 4단계: 선언형 Base 생성

```python
Base = declarative_base()
```

모든 SQLAlchemy ORM 모델은 이 `Base`를 상속해야 한다. Alembic은 `Base.metadata`를 읽어 모델과 데이터베이스 구조의 차이를 찾는다.

SQLAlchemy 2.x 스타일로 정리할 때는 다음 방식도 사용할 수 있다.

```python
from sqlalchemy.orm import DeclarativeBase


class Base(DeclarativeBase):
    pass
```

다만 팀 프로젝트에서는 중간에 한 방식으로 갑자기 바꾸지 말고 팀원과 함께 변경 범위를 확인해야 한다.

#### 5단계: FastAPI 의존성 함수

```python
async def async_get_db():
    async with AsyncSessionLocal() as db:
        yield db
```

`yield` 앞에서는 세션을 생성하고, 요청 처리가 끝나면 `async with`가 세션을 닫는다.

API에서는 다음처럼 주입받는다.

```python
from fastapi import Depends
from sqlalchemy.ext.asyncio import AsyncSession

from app.core.db.databases import async_get_db


async def example_api(
    db: AsyncSession = Depends(async_get_db),
):
    ...
```

세션을 닫는 것과 트랜잭션을 commit하는 것은 다른 일이다. 현재 `async_get_db`는 세션 생명주기만 관리하므로 데이터 변경 시 Service 또는 Repository에서 `commit()`과 오류 발생 시 `rollback()`을 처리해야 한다.

## 5. SQLAlchemy 모델 작성과 Alembic 마이그레이션

### 5.1 전체 작업 순서

```text
1. SQLAlchemy 모델 작성
2. app/models/__init__.py에서 모델 import
3. .env와 데이터베이스 실행 상태 확인
4. Alembic revision 자동 생성
5. 생성된 migration 파일 검토
6. upgrade로 데이터베이스 반영
7. 실제 테이블과 현재 revision 확인
```

### 5.2 모델 작성

예를 들어 다음 파일을 만든다.

```text
app/models/user.py
```

모델은 반드시 `app.core.db.databases.Base`를 상속해야 한다.

```python
class User(TimestampMixin, Base):
    __tablename__ = "users"
    ...
```

### 5.3 모델 import 등록

```python
# app/models/__init__.py
from app.models.user import User

__all__ = ["User"]
```

현재 `alembic/env.py`는 `from app import models`를 실행하고 `Base.metadata`를 `target_metadata`로 사용한다. 따라서 `models/__init__.py`에서 모델이 import되어야 테이블 정보가 metadata에 등록된다.

### 5.4 마이그레이션 파일 자동 생성

프로젝트 루트에서 실행한다.

```bash
uv run alembic revision --autogenerate -m "create users table"
```

성공하면 `alembic/versions/`에 새 Python 파일이 생성된다.

### 5.5 생성 파일 검토

자동 생성 결과를 바로 적용하지 말고 다음 내용을 확인한다.

- `upgrade()`에 원하는 테이블과 컬럼이 생성되는지
- `downgrade()`에 되돌리는 코드가 있는지
- 컬럼 길이와 nullable 설정이 맞는지
- unique index와 foreign key가 맞는지
- 기존 데이터가 있는 테이블에 위험한 변경이 없는지
- 의도하지 않은 테이블 삭제가 포함되지 않았는지

Alembic의 `--autogenerate`는 모델과 데이터베이스의 차이를 바탕으로 초안을 만드는 기능이다. 생성 결과가 항상 완전하거나 안전한 것은 아니므로 사람이 검토해야 한다.

### 5.6 데이터베이스에 반영

```bash
uv run alembic upgrade head
```

현재 적용된 revision을 확인한다.

```bash
uv run alembic current
```

마이그레이션 이력을 확인한다.

```bash
uv run alembic history
```

마지막 마이그레이션을 한 단계 되돌린다.

```bash
uv run alembic downgrade -1
```

공유 데이터베이스에서 downgrade를 실행하면 데이터가 삭제될 수 있으므로 팀원과 상의 없이 실행하지 않는다.

### 5.7 Alembic 연결 구조

현재 `alembic/env.py`는 다음 작업을 수행한다.

```text
config.py에서 환경 변수 로드
    ↓
databases.py에서 DATABASE_URL 생성
    ↓
alembic/env.py가 DATABASE_URL을 alembic 설정에 반영
    ↓
app.models import
    ↓
Base.metadata를 target_metadata로 지정
    ↓
모델과 실제 DB 구조 비교
```

`alembic.ini`의 `sqlalchemy.url`은 주석 처리되어 있고, 실제 URL은 `alembic/env.py`에서 `DATABASE_URL`로 덮어쓴다. 따라서 Alembic도 애플리케이션과 같은 `.env` 설정을 사용한다.

## 6. API 구현 흐름

새로운 기능을 구현할 때는 데이터베이스에서 API 방향으로 한 단계씩 작성하면 구조를 확인하기 쉽다.

### 6.1 권장 구현 순서

#### 1단계: Model

데이터베이스에 저장할 구조를 작성한다.

```text
app/models/user.py
```

#### 2단계: Alembic migration

모델 변경을 데이터베이스 스키마에 반영한다.

```bash
uv run alembic revision --autogenerate -m "create users table"
uv run alembic upgrade head
```

#### 3단계: Schema

요청과 응답 형식을 작성한다.

```text
app/schemas/user.py
```

예:

- `UserCreate`
- `UserUpdate`
- `UserResponse`

#### 4단계: Repository

SQLAlchemy를 사용한 조회와 저장 코드를 작성한다.

```text
app/repositories/user_repository.py
```

#### 5단계: Service

중복 검사와 같은 업무 규칙을 작성하고 트랜잭션을 관리한다.

```text
app/services/user_service.py
```

#### 6단계: API Router

URL, HTTP Method, 상태 코드, 의존성 주입을 작성한다.

```text
app/apis/users.py
```

#### 7단계: Router 등록

`app/main.py`에서 라우터를 등록한다.

```python
from app.apis import users

app.include_router(users.router)
```

#### 8단계: 테스트

서버를 실행한다.

```bash
uv run fastapi dev app/main.py
```

Swagger UI에 접속한다.

```text
http://127.0.0.1:8000/docs
```

다음 항목을 확인한다.

- 정상 요청
- 필수값 누락
- 잘못된 자료형
- 존재하지 않는 ID
- 중복 데이터
- 권한이 필요한 기능
- 데이터베이스 반영 여부
- 응답에 비밀번호 같은 민감 정보가 포함되지 않는지

### 6.2 요청 한 건이 처리되는 과정

회원 생성 API를 예로 들면 다음과 같다.

```text
1. 클라이언트가 POST 요청 전송
2. API가 UserCreate Schema로 Request Body 검증
3. API가 Depends(async_get_db)로 AsyncSession 주입
4. API가 UserService 호출
5. Service가 Repository를 통해 이메일 중복 조회
6. Service가 비밀번호 해시와 회원 생성 규칙 처리
7. Repository가 SQLAlchemy 모델을 세션에 추가
8. Service가 commit
9. SQLAlchemy 모델 반환
10. UserResponse Schema가 응답 필드 제한
11. FastAPI가 JSON 응답 반환
```

이 흐름에서 각 계층은 자기 역할만 담당해야 한다.

- API: HTTP 요청과 응답
- Schema: 데이터 형식과 검증
- Service: 업무 규칙
- Repository: 데이터베이스 접근
- Model: 테이블 구조
- Core: 공통 설정과 연결 기반

## 7. 현재 템플릿에서 확인한 주의사항

### 7.1 `app/models/__init__.py`가 비어 있음

첫 SQLAlchemy 모델을 추가할 때 모델 import도 함께 작성해야 Alembic 자동 생성에서 누락되지 않는다.

### 7.2 `app/Dockerfile`이 비어 있음

`docker-compose.yml`은 `app/Dockerfile`을 사용해 FastAPI 이미지를 빌드하도록 작성되어 있다. 현재 Dockerfile이 비어 있으므로 Docker Compose 실행 전 Python, uv, 프로젝트 파일 복사, 의존성 설치 등을 포함한 Dockerfile 작성이 필요하다.

### 7.3 정적 파일 경로가 안내 구조와 다름

과제 안내에는 `app/static`, `app/media`가 표시되어 있지만 현재 `app/main.py`는 프로젝트 루트의 `static`, `media`를 사용한다. 실제 저장소에도 루트 `static/index.html`이 존재한다.

경로를 바로 변경하면 프론트엔드가 깨질 수 있으므로 팀에서 한 기준을 선택한 뒤 `main.py`, Docker volume, 파일 위치를 함께 수정해야 한다.

### 7.4 Docker 환경에서는 `DB_HOST=mysql` 필요

컨테이너 내부의 `localhost`는 같은 컨테이너 자신을 가리킨다. FastAPI 컨테이너에서 MySQL 서비스로 접속하려면 Compose 서비스 이름인 `mysql`을 사용한다.

### 7.5 데이터베이스 비밀번호를 코드 기본값에 의존하지 않기

개발 편의를 위한 기본값이 있지만 팀별 `.env`를 작성하고 민감한 값은 저장소에 올리지 않는다.

## 8. 팀 작업 시 파일 배치 기준

기능 하나를 추가할 때 다음처럼 같은 기능 이름을 사용하면 찾기 쉽다.

```text
app/
├── models/
│   └── user.py
├── schemas/
│   └── user.py
├── repositories/
│   └── user_repository.py
├── services/
│   └── user_service.py
└── apis/
    └── users.py
```

파일 이름과 클래스 이름은 팀에서 규칙을 통일한다.

예:

- 모델: `User`
- Schema: `UserCreate`, `UserUpdate`, `UserResponse`
- Repository: `UserRepository`
- Service: `UserService`
- Router 파일: `users.py`

순환 import가 생기지 않도록 의존 방향도 통일한다.

```text
apis → services → repositories → models
apis → schemas
services → schemas, repositories, models
repositories → models
models → core.db
```

하위 계층이 상위 계층을 import하지 않도록 주의한다.

## 9. 이번 학습 내용 정리

이 템플릿은 FastAPI, Pydantic, SQLAlchemy, Alembic, MySQL을 사용해 계층형 웹 API를 구현하도록 구성되어 있다.

핵심은 다음과 같다.

1. `core`는 환경 설정과 데이터베이스 연결 같은 공통 기반을 담당한다.
2. `models`는 데이터베이스 테이블 구조를 정의한다.
3. `schemas`는 API 요청과 응답의 데이터 형식을 검증한다.
4. `repositories`는 SQLAlchemy를 사용한 데이터 접근을 담당한다.
5. `services`는 중복 확인, 트랜잭션 같은 업무 규칙을 담당한다.
6. `apis`는 HTTP 요청과 응답을 처리하고 Service를 호출한다.
7. `main.py`에서 FastAPI 앱과 라우터를 연결한다.
8. `config.py`는 `.env`를 읽고 `databases.py`가 비동기 엔진과 세션을 만든다.
9. SQLAlchemy 모델은 `Base`를 상속하고 `app/models/__init__.py`에서 import해야 Alembic이 발견할 수 있다.
10. Alembic 자동 생성 파일은 적용 전에 반드시 직접 검토한다.
11. `pyproject.toml`은 직접 의존성을 선언하고 `uv.lock`은 실제 설치 버전을 고정한다.
12. 팀원은 기능 브랜치에서 작업한 뒤 Pull Request와 Review를 거쳐 `develop`, 마지막으로 `main`에 병합한다.

## 10. 문서 작업의 Git 흐름

우리 팀의 브랜치 전략을 적용하면 이번 문서는 다음 순서로 반영한다.

```text
develop 최신화
    ↓
feature/day3-project-template-analysis 생성
    ↓
docs/3일차_프로젝트_뜯어보기.md 작성
    ↓
Add & Commit
    ↓
Push
    ↓
feature → develop Pull Request
    ↓
팀원 Review 후 Merge
    ↓
develop 통합 확인
    ↓
develop → main Pull Request
    ↓
최종 Review 후 Merge
```

예시 명령은 다음과 같다.

```bash
git switch develop
git pull origin develop
git switch -c feature/day3-project-template-analysis

git add "docs/3일차_프로젝트_뜯어보기.md"
git commit -m "docs: 프로젝트 템플릿 구조 정리"
git push -u origin feature/day3-project-template-analysis
```

첫 번째 Pull Request 방향은 다음과 같다.

```text
base: develop
compare: feature/day3-project-template-analysis
```

통합 확인 후 최종 Pull Request 방향은 다음과 같다.

```text
base: main
compare: develop
```

`main`과 `develop`에 직접 문서를 커밋하지 않고, 팀원이 작성 내용을 확인할 수 있도록 Pull Request 과정을 남긴다.
