## 0. 개요

| 항목 | 내용 |
| --- | --- |
| 대상 도메인 | User(회원) — 회원가입, 인증(로그인/로그아웃/토큰), 회원 정보 관리, 권한 관리 |
| 배경 | 폐렴 환자 관리 백오피스(AI Health) 서비스의 내부 직원(의료/개발/연구 부서) 계정 관리 |

## 1. 액터(사용자 유형) 정의

`app/models/enums.py`의 `RoleEnum` 기준으로 3종의 역할이 있으며, 별도의 비로그인 사용자를 포함하면 총 4가지 액터가 있습니다.

| 액터 | role 값 | 설명 |
| --- | --- | --- |
| 비로그인 사용자 (Guest) | - | 아직 계정이 없거나 로그인하지 않은 사용자. 회원가입/로그인만 가능 |
| 대기자 (Pending) | `PENDING` | 회원가입 직후 기본 부여되는 권한. 자기 정보 조회/수정, 비밀번호 변경, 탈퇴 가능 |
| 스태프 (Staff) | `STAFF` | Admin이 승인하여 부여하는 일반 직원 권한. Pending과 동일한 기능 + (추후 업무 도메인 기능 접근) |
| 관리자 (Admin) | `ADMIN` | 회원 목록 조회, 회원 권한 변경 등 회원 관리 기능에 접근 가능 |

> Pending/Staff 간 기능 차이(예: 특정 화면 접근 제한)는 이번 User 도메인 요구사항 범위에는 명시되어 있지 않으며, 다른 도메인(환자/진료기록 등) 설계 시 추가로 정의가 필요합니다.

## 2. 용어 정의

| 용어 | 값 | 설명 |
| --- | --- | --- |
| `gender` | `M`, `F` | 성별 |
| `department` | `MEDICAL`, `DEV`, `RESEARCH` | 소속 부서 (의료 / 개발 / 연구) |
| `role` | `PENDING`, `STAFF`, `ADMIN` | 회원 권한 (대기자 / 스태프 / 어드민) |

| refresh token | - | access token 재발급에 사용하는 장기 토큰. 서버가 `httpOnly` 쿠키로만 내려줌 |

## 4. 상세 기능 요구사항

### REQ-USER-001 회원가입

- **설명**: Guest가 이메일/비밀번호/이름/부서/성별/휴대폰번호를 입력하여 계정을 생성한다.
- **입력 항목**

| 항목 | 필수 | 제약 조건 |
| --- | --- | --- |
| 이메일 | Y | 고유값, 최대 255자, 이메일 형식 |
| 비밀번호 | Y | 8~20자, 대문자·소문자·특수문자 포함 |
| 이름 | Y | 최대 20자 |
| 부서 | Y | `MEDICAL` / `DEV` / `RESEARCH` 중 하나 |
| 성별 | Y | `M` / `F` |
| 휴대폰번호 | Y | 고유값 |

- **처리 조건**
  - 가입 시 `role`은 사용자가 직접 지정할 수 없고, 서버가 항상 `PENDING`으로 고정한다 (권한 상승 방지).
  - `is_active`는 기본값 `true`로 생성한다.
  - 비밀번호는 평문으로 저장하지 않고 해싱하여 저장한다.
- **예외 상황**
  - 이메일 또는 휴대폰번호가 이미 등록되어 있으면 가입을 거부한다.
  - 필수 입력값이 비어있거나 형식(이메일/비밀번호/휴대폰번호)이 올바르지 않으면 가입을 거부한다.

### REQ-USER-002 로그인 (관련: NFR-USER-001)

- **설명**: Guest가 이메일/비밀번호로 인증에 성공하면 서비스 이용을 위한 토큰을 발급받는다.
- **입력 항목**: 이메일, 비밀번호
- **처리 조건**
  - 이메일·비밀번호가 일치하고 `is_active=true`인 계정만 로그인을 허용한다.
  - 로그인 성공 시 access token과 refresh token을 발급한다 (토큰 정책은 NFR-USER-001 참고).
- **예외 상황**
  - 이메일 또는 비밀번호가 일치하지 않으면 로그인을 거부한다.
  - 비활성화(`is_active=false`)된 계정은 로그인을 거부한다.

### REQ-USER-003 로그아웃

- **설명**: 로그인한 사용자가 화면 헤더의 로그아웃 버튼을 통해 현재 세션을 종료한다.
- **처리 조건**: 로그아웃 성공 시 서버에 저장/전달된 인증 상태(refresh token 쿠키)를 즉시 만료시킨다.
- **후속 동작**: 로그아웃 성공 시 프론트엔드는 로그인 페이지로 전환한다.

### REQ-USER-004 회원 목록 조회

- **설명**: Admin이 전체 회원을 조건에 따라 검색·필터링하여 목록으로 조회한다.
- **조회 조건**: 이메일/이름 검색어, 부서(department) 필터
- **목록에서 확인 가능한 항목**: 이메일, 이름, 부서, 성별, 휴대폰번호, 활성화 여부
- **처리 조건**: Admin 권한을 가진 사용자만 조회할 수 있다.
- **예외 상황**: Admin이 아닌 사용자가 조회를 시도하면 거부한다.

### REQ-USER-005 회원 권한 변경

- **설명**: Admin이 회원 목록에서 특정 대상자를 선택하여 권한(대기자/스태프/어드민)을 변경한다.
- **처리 조건**
  - Admin 권한을 가진 사용자만 다른 회원의 권한을 변경할 수 있다.
  - 변경 가능한 값은 `PENDING` / `STAFF` / `ADMIN` 3가지이다.
  - 본인 스스로의 권한은 이 기능이 아닌 별도 논의 대상으로 하며, 이 요구사항 자체에는 자기 자신 변경 제한이 명시되어 있지 않다 (실 구현 전 팀 논의 필요).
- **예외 상황**
  - Admin이 아닌 사용자가 시도하면 거부한다.
  - 대상 회원이 존재하지 않으면 실패 처리한다.

### REQ-USER-006 마이페이지 조회

- **설명**: 로그인한 사용자가 본인의 회원 정보를 조회한다.
- **조회 가능 항목**: 이메일, 이름, 부서, 성별, 휴대폰번호, 권한(role)
- **처리 조건**: 반드시 로그인한 "본인"의 정보만 조회할 수 있어야 하며, 다른 회원의 정보를 조회할 수 없어야 한다.

### REQ-USER-007 회원 정보 수정

- **설명**: 로그인한 사용자가 본인의 회원 정보 중 일부를 수정한다.
- **수정 가능 항목**: 부서, 휴대폰번호
- **수정 불가 항목**: 이메일, 이름, 성별, 권한(role) — 이 요구사항으로는 변경할 수 없다.
- **처리 조건**
  - 최소 1개 이상의 항목을 포함해야 한다 (빈 요청 불가).
  - 변경하려는 휴대폰번호가 다른 회원과 중복되면 거부한다.

### REQ-USER-008 비밀번호 변경 (관련: NFR-USER-002)

- **설명**: 로그인한 사용자가 기존 비밀번호를 검증받은 뒤 새로운 비밀번호로 변경한다.
- **입력 항목**: 기존 비밀번호, 새 비밀번호(8~20자, 대/소문자+특수문자 포함 — 회원가입과 동일한 비밀번호 정책)
- **처리 조건**: 입력한 기존 비밀번호가 실제 저장된 비밀번호와 일치해야 변경을 허용한다.
- **예외 상황**
  - 기존 비밀번호가 일치하지 않으면 거부한다.
  - 새 비밀번호가 정책(길이/구성)에 맞지 않으면 거부한다.

### REQ-USER-009 회원 탈퇴

- **설명**: 로그인한 사용자가 본인의 계정을 탈퇴 처리한다.
- **처리 조건**: 탈퇴 시 관련 회원 데이터를 즉시 삭제(hard delete)한다.
- **미결 사항**: 다른 테이블에서 FK로 참조 중인 데이터(예: `xray_images.uploader_id`)의 처리 방식(연쇄 삭제 vs NULL 처리)은 이 문서 작성 시점에 확정되지 않았으며, 실제 구현 전 팀 논의가 필요하다.

## 5. 비기능 요구사항

### NFR-USER-001 인증 토큰 정책

- access token 만료 시간: 30분
- refresh token 만료 시간: 7일
- refresh token은 클라이언트 JavaScript에서 접근할 수 없도록 `httpOnly` 쿠키로만 전달한다 (응답 바디에 포함하지 않는다).
- access token 만료 시, refresh token으로 재발급을 시도하고 refresh token까지 만료/무효한 경우 재로그인을 유도한다.
- 토큰 페이로드에는 최소 식별 정보(`user_id`)만 포함한다.

### NFR-USER-002 비밀번호 입력 보안

- 비밀번호 입력창은 기본적으로 마스킹 처리한다.
- 사용자가 "보기" 아이콘을 클릭했을 때만 평문으로 노출한다.

### NFR-USER-003 API 응답 성능

- 회원 관련 API는 3초 이내에 응답해야 한다.
- (참고) 회원 수가 많아질 것을 대비해 회원 목록 조회에는 페이지네이션 적용을 권장한다.

## 6. 데이터 제약 조건 (참고: `app/models/user.py`)

| 필드 | 제약 |
| --- | --- |
| `email` | unique, not null |
| `hashed_password` | not null (평문 비밀번호는 저장하지 않음) |
| `name` | not null |
| `department` | not null, enum(`MEDICAL`/`DEV`/`RESEARCH`) |
| `gender` | not null, enum(`M`/`F`) |
| `phone_number` | unique, not null |
| `role` | not null, enum(`PENDING`/`STAFF`/`ADMIN`), 기본값 `PENDING` |
| `is_active` | not null, 기본값 `true` |
