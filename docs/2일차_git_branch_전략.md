# 2일차 Git Branch 전략

## 1. 브랜치 전략의 목적

Git 브랜치 전략은 여러 명의 개발자가 서로의 작업에 직접 영향을 주지 않고 기능을 개발하기 위한 협업 규칙이다.

각 팀원은 별도의 브랜치에서 작업하고, 작업이 완료되면 Pull Request와 코드 리뷰를 거쳐 기준 브랜치에 병합한다. 이를 통해 작업 이력을 확인하고 코드 충돌과 오류가 기준 브랜치에 반영되는 것을 줄일 수 있다.

## 2. Git Flow와 GitHub Flow 비교

### Git Flow

Git Flow는 다음과 같이 여러 종류의 브랜치를 운영한다.

- main: 정식 배포 버전
- develop: 다음 배포를 준비하는 개발 버전
- feature/*: 기능 개발
- release/*: 배포 준비
- hotfix/*: 배포된 버전의 긴급 수정

Git Flow는 개발 단계와 출시 단계를 명확하게 분리할 수 있다. 반면 브랜치 종류가 많고 병합 절차가 복잡하므로, 짧은 프로젝트에서는 관리 부담이 커질 수 있다.

### GitHub Flow

GitHub Flow는 main과 작업 브랜치를 중심으로 운영한다.

개발자는 최신 main에서 작업 브랜치를 생성하고, 기능 구현과 테스트가 끝나면 Pull Request를 생성한다. 팀원의 리뷰를 받은 뒤 작업 브랜치를 main에 병합한다.

GitHub Flow는 구조가 간단하고 작업 단위가 명확하여 짧은 기간에 여러 명이 협업하는 프로젝트에 적합하다.

## 3. 우리 팀의 선택: GitHub Flow

우리 팀은 GitHub Flow를 사용한다.

이 프로젝트는 단기 프로젝트이며, 별도의 정식 출시 버전이나 release 브랜치를 관리하지 않는다.

따라서 여러 장기 브랜치를 운영하는 Git Flow보다 main과 작업 브랜치를 중심으로 운영하는 GitHub Flow가 현재 프로젝트에 적합하다고 판단하였다.

## 4. 기준 브랜치

### main

main 브랜치에는 로컬 실행, Swagger 테스트, Pull Request 리뷰가 완료된 코드만 병합한다.

팀원은 main 브랜치에서 직접 코드나 문서를 수정하지 않는다.

## 5. 브랜치 이름 규칙

### 기능 개발

feature/기능명

### 문서 작성

docs/문서명

### 오류 수정

fix/오류명

## 6. Stage 2 완료 조건

- docs/2일차_git_branch_전략.md가 원격 저장소의 main 브랜치에 존재
- 팀원별 작업 브랜치가 GitHub에서 확인
- 팀원별 commit과 push 기록 존재
- 각 작업에 대한 Pull Request 존재
- 작성자 외 팀원의 리뷰 기록 존재
- 다섯 개 API가 main에 병합
- FastAPI 서버가 로컬에서 실행
- Swagger UI에서 다섯 개 API가 정상적으로 표시
- 정상 요청과 주요 예외 요청이 요구사항에 맞게 동작
