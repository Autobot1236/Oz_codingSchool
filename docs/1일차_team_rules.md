# Oz_codingSchool Team Rules

## Team Rules

### 1. 코어 타임

매일 수업 이후 팀원들이 가능한 시간을 공유하고, 최소 30분 이상 진행 상황을 함께 확인합니다.

### 2. 1일 1커밋

각자 맡은 작업은 자신의 브랜치에서 진행하고, 하루에 최소 1번 이상 의미 있는 단위로 커밋합니다.

### 3. 작업 상황 공유

작업 시작 전 오늘 할 일을 공유하고, 작업 종료 후 완료한 내용과 막힌 부분을 팀 채널에 남깁니다.

### 4. 부드러운 커뮤니케이션

코드 리뷰와 회의에서는 상대방을 비난하지 않고, 문제 상황과 개선 방향을 중심으로 이야기합니다.

### 5. 모르는 내용은 빠르게 질문

혼자 오래 막히지 않고, 30분 이상 해결되지 않는 문제는 팀원에게 공유하여 함께 해결합니다.

## Git & GitHub Rules

### 1. 브랜치 전략

- `main`: 최종적으로 동작이 확인된 코드만 반영합니다.
- `feature/기능명`: 개인 기능 개발 브랜치로 사용합니다.
- `fix/수정명`: 오류 수정 브랜치로 사용합니다.

### 2. 커밋 메시지 규칙

- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정
- `style`: 코드 포맷팅 및 스타일 수정
- `refactor`: 기능 변화 없는 코드 구조 개선
- `chore`: 환경 설정 및 기타 작업

예시:

```text
feat: 환자 목록 조회 API 추가
docs: 1일차 팀 규칙 작성
fix: 로그인 요청 오류 수정
```

### 3. Pull Request 규칙

`main` 브랜치에 직접 작업하지 않고, 기능 브랜치에서 작업한 뒤 Pull Request를 생성합니다.

PR에는 작업 내용, 테스트 여부, 리뷰가 필요한 부분을 간단히 작성합니다.

### 4. Pull 전 확인

작업을 시작하기 전에는 항상 최신 코드를 가져옵니다.

```bash
git pull origin main
```

### 5. 충돌 해결

충돌이 발생하면 혼자 임의로 삭제하지 않고, 해당 파일을 작업한 팀원과 확인 후 해결합니다.

## 추가 협업 규칙

### 6. 역할과 담당 작업 관리

각 작업에는 주 담당자와 검토 담당자를 지정합니다.

주 담당자는 코드 작성과 테스트를 수행하고, 검토 담당자는 요구사항 충족 여부와 다른 기능에 미치는 영향을 확인합니다.

담당자가 작업을 완료하기 어렵거나 일정이 지연될 것으로 예상되는 경우, 마감 직전이 아니라 확인 즉시 팀 채널에 공유합니다.

### 7. 공용 파일 수정 전 공유

다음과 같이 여러 기능에 영향을 미치는 공용 파일을 수정할 때는 작업 시작 전에 팀 채널에 공유합니다.

* `app/main.py`
* `app/core/`
* `app/core/db/`
* `app/models/`
* `app/schemas/`의 공통 스키마
* `docker-compose.yml`
* `pyproject.toml`
* `.gitignore`

동일한 파일을 여러 명이 동시에 수정해야 하는 경우 작업 범위를 먼저 나누거나, 한 명이 통합 작업을 담당합니다.

### 8. 일정 지연 및 부재 공유

수업 불참, 조퇴, 외출 또는 개인 일정으로 작업 참여가 어려운 경우 가능한 한 사전에 팀 채널에 공유합니다.

담당 작업이 예정된 시간까지 완료되지 않을 경우 다음 내용을 함께 공유합니다.

* 현재까지 완료한 내용
* 남은 작업
* 발생한 문제
* 다른 팀원의 도움이 필요한 부분
* 예상 완료 시점

## Git & GitHub 추가 규칙

### 6. 브랜치 이름 작성 기준

브랜치 이름은 작업 목적이 확인되도록 작성합니다.

```text
feature/user-signup
feature/patient-list-api
feature/pneumonia-prediction
fix/login-token-error
docs/stage3-project-structure
test/patient-api
```

하나의 브랜치에서는 가능한 한 하나의 기능이나 하나의 문서만 작업합니다.

### 7. Pull Request 승인 및 병합 기준

Pull Request는 최소 한 명 이상의 팀원이 검토한 후 병합합니다.

PR 작성자는 특별한 사유가 없는 한 자신의 PR을 검토 없이 바로 병합하지 않습니다.

다음 조건을 모두 충족한 경우에만 `main` 또는 `develop` 브랜치로 병합합니다.

* 코드가 로컬 환경에서 실행됨
* 관련 API가 Swagger UI에서 정상적으로 확인됨
* 정상 요청과 주요 예외 요청을 테스트함
* 불필요한 디버깅 코드와 출력문을 제거함
* 충돌이 해결됨
* 관련 문서가 필요한 경우 함께 수정됨
* 리뷰 의견이 반영되거나 팀원 간 합의가 완료됨

### 8. Pull Request 작성 형식

PR에는 다음 내용을 작성합니다.

```text
## 작업 내용
- 구현하거나 수정한 내용

## 관련 요구사항
- 해당 Stage 또는 요구사항 ID

## 테스트 방법
- 실행한 명령어
- Swagger UI 또는 화면에서 확인한 결과

## 영향 범위
- 변경으로 영향을 받을 수 있는 API, 화면 또는 데이터베이스

## 확인이 필요한 부분
- 리뷰어가 중점적으로 확인해야 할 내용
```

PR의 범위가 지나치게 커지지 않도록 기능, 문서, 설정 변경은 가능한 한 분리합니다.

### 9. 작업 완료 기준

작업은 커밋 또는 코드 작성만으로 완료 처리하지 않습니다.

API 작업은 다음 조건을 충족해야 완료로 판단합니다.

* 서버가 정상 실행됨
* Swagger UI에 API가 표시됨
* 정상 요청이 예상한 응답을 반환함
* 잘못된 입력과 존재하지 않는 데이터에 대한 예외 처리가 작동함
* 비밀번호와 토큰 등 민감정보가 응답에 포함되지 않음

문서 작업은 파일 경로와 파일명이 과제 요구사항과 일치하고, PR을 통해 목표 브랜치에 병합된 경우 완료로 판단합니다.

프론트엔드 작업은 화면 표시뿐 아니라 실제 API 호출과 오류 상황을 확인한 경우 완료로 판단합니다.

### 10. API 명세 변경 규칙

이미 공유된 API의 다음 항목을 변경할 때는 관련 담당자에게 먼저 알리고 합의한 후 수정합니다.

* HTTP Method
* Endpoint
* Path Parameter와 Query Parameter
* Request Body
* Response Body
* Status Code
* 데이터 필드명과 자료형

API 명세가 변경되면 관련 문서와 프론트엔드 호출 코드도 함께 수정합니다.

### 11. 보안 및 개인정보 규칙

다음 정보는 GitHub에 커밋하지 않습니다.

* `.env` 파일
* 데이터베이스 비밀번호
* JWT Secret Key
* API Key
* 개인 액세스 토큰
* 실제 환자의 이름, 연락처, 진료기록 및 X-Ray 이미지

개발과 테스트에는 가상 환자 정보와 과제에서 제공되거나 사용이 허용된 이미지만 사용합니다.

환경변수 예시는 실제 값을 제외한 `.env.example` 파일로 공유합니다.

```text
DATABASE_URL=
JWT_SECRET_KEY=
ACCESS_TOKEN_EXPIRE_MINUTES=
```

민감정보를 실수로 커밋한 경우 파일만 삭제하지 않고 즉시 팀원에게 공유하며, 노출된 키와 비밀번호를 교체합니다.

### 12. 충돌 방지와 최신화

새 브랜치를 만들기 전 `main` 또는 `develop` 브랜치를 최신 상태로 갱신합니다.

```bash
git switch main
git pull origin main
git switch -c feature/기능명
```

장기간 작업한 브랜치는 PR 생성 전에 기준 브랜치의 최신 변경사항을 반영하고 충돌 여부를 확인합니다.

충돌 해결 과정에서 다른 팀원의 코드를 임의로 삭제하거나 이전 상태로 되돌리지 않습니다.

### 13. 코드 리뷰 원칙

코드 리뷰에서는 작성자를 평가하지 않고 코드의 동작, 요구사항, 유지보수성, 보안과 예외 처리를 확인합니다.

리뷰 의견에는 가능한 한 문제의 이유와 수정 방향을 함께 작성합니다.

필수 수정이 필요한 의견과 선택적으로 개선할 수 있는 의견을 구분합니다.

```text
[필수] 존재하지 않는 환자 ID에 대해 404 처리가 필요합니다.
[제안] 중복 코드를 함수로 분리하면 읽기 쉬워질 것 같습니다.
[질문] 이 필드를 Optional로 정의한 이유를 확인하고 싶습니다.
```

### 14. 일일 통합 점검

매일 작업 종료 전 다음 항목을 함께 확인합니다.

* 당일 병합된 PR
* 병합 대기 중인 PR
* 해결되지 않은 오류
* 다음 날 작업 담당자
* API와 데이터베이스 구조 변경사항
* 전체 애플리케이션 실행 여부

개별 기능이 정상 작동하더라도 다른 기능과 연결했을 때 오류가 발생할 수 있으므로, 주요 Stage 종료 시에는 팀원 모두가 통합 실행 결과를 확인합니다.
