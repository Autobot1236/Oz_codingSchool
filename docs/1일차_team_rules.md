# Oz_codingSchool Team Rules

## Team Rules

### 1. 코어 타임

매일 수업 이후 팀원들이 가능한 시간을 공유하고, 최소 30분 이상 진행 상황을 함께 확인합니다.

### 2. 1일 1커밋

각자 맡은 작업은 자신의 브랜치에서 진행하고, 하루에 최소 1번 이상 의미 있는 단위로 커밋합니다.

### 3. 작업 상황 공유

작업 시작 전 오늘 할 일을 공유하고, 작업 종료 후 완료한 내용과 막힌 부분을 팀 채널에 남깁니다.

### 4. 부드러운 커뮤니케이션

코드 리뷰와 회의에서는 상대방을 비난하지 않고, 문제 상황과 개선 방향을 중심으로 이야기합니다.

### 5. 모르는 내용은 빠르게 질문

혼자 오래 막히지 않고, 30분 이상 해결되지 않는 문제는 팀원에게 공유하여 함께 해결합니다.

## Git & GitHub Rules

### 1. 브랜치 전략

- `main`: 최종적으로 동작이 확인된 코드만 반영합니다.
- `feature/기능명`: 개인 기능 개발 브랜치로 사용합니다.
- `fix/수정명`: 오류 수정 브랜치로 사용합니다.

### 2. 커밋 메시지 규칙

- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정
- `style`: 코드 포맷팅 및 스타일 수정
- `refactor`: 기능 변화 없는 코드 구조 개선
- `chore`: 환경 설정 및 기타 작업

예시:

```text
feat: 환자 목록 조회 API 추가
docs: 1일차 팀 규칙 작성
fix: 로그인 요청 오류 수정
```

### 3. Pull Request 규칙

`main` 브랜치에 직접 작업하지 않고, 기능 브랜치에서 작업한 뒤 Pull Request를 생성합니다.

PR에는 작업 내용, 테스트 여부, 리뷰가 필요한 부분을 간단히 작성합니다.

### 4. Pull 전 확인

작업을 시작하기 전에는 항상 최신 코드를 가져옵니다.

```bash
git pull origin main
```

### 5. 충돌 해결

충돌이 발생하면 혼자 임의로 삭제하지 않고, 해당 파일을 작업한 팀원과 확인 후 해결합니다.
